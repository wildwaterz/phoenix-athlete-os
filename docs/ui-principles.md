# UI Principles

Version: 1.0  
Owner: Kevin Sauvageau  
Product: Project Phoenix OS  
Status: Living Document

## Purpose

This document defines the visual and interaction design principles for Project Phoenix OS.

## Design Identity

Project Phoenix should feel like:

- Mission control
- A premium athletic performance platform
- A coach-supported operating system

It should not feel like:

- A hospital portal
- A generic habit tracker
- A cluttered medical form
- A gamified toy

## Visual Inspiration

- WHOOP
- Garmin Connect
- Apple Health
- Linear

## Core UI Principles

### 1. One primary question per screen

Each screen should answer one main question.

Dashboard:

“What should I do today?”

Check-In:

“What evidence am I capturing?”

Missions:

“What am I working toward?”

Milestones:

“What have I earned?”

### 2. Mission over checklist

The athlete should feel like they are progressing through a campaign, not filling paperwork.

### 3. Calm, not clinical

Use reassuring language and avoid scary visual patterns unless truly necessary.

### 4. Trends over snapshots

Whenever possible, show whether a metric is improving, worsening, or stable.

### 5. Explain why

Important cards should explain why they matter.

### 6. Reduce cognitive load

Group forms by how athletes think:

- Knee status
- Movement and control
- Recovery inputs
- Notes

### 7. Make export easy

Coach Packet export should require one click to copy.

### 8. Mobile matters

Daily check-ins will often happen on a phone.

The mobile experience should be as important as desktop.

## Tone

The product voice should be:

- Calm
- Direct
- Encouraging
- Evidence-based
- Occasionally motivational

Avoid:

- Hype language
- Medical fear language
- Overly robotic phrasing
- Overpromising
